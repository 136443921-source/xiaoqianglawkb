## 🎯 系统概述

### 什么是"小强律师数字分身系统"？

**小强**是一个专为律师设计的AI数字分身，具有以下核心特征：

- **直接、有观点、不废话**：拒绝客套话，直接给结论和依据
- **偏技术分析**：注重法条精准解读、判例深度分析、裁判规则提炼
- **实战导向**：关注法律实务中的裁判规则、证据认定和法律适用问题
- **持续进化**：通过知识库不断积累和沉淀专业知识

### 系统架构

```
┌─────────────────────────────────────────────────────┐
│         小强律师数字分身系统                     │
├─────────────────────────────────────────────────────┤
│  Layer 1: 交互层                               │
│  - WorkBuddy 对话界面                          │
│  - Obsidian 知识展示                           │
├─────────────────────────────────────────────────────┤
│  Layer 2: 功能层                               │
│  - 法律知识库（LLM Wiki）                     │
│  - 法条检索与适用分析                         │
│  - 案例检索与裁判规则提炼                     │
│  - 文书撰写辅助                               │
├─────────────────────────────────────────────────────┤
│  Layer 3: 数据层                               │
│  - Raw 层：原始资料（只读）                  │

└─────────────────────────────────────────────────────┘
```

### 核心优势

| 特性 | 说明 | 适用场景 |
|------|------|----------|
| **知识沉淀** | 一次编译，持续复用 | 长期积累专业知识和经验 |
| **自动关联** | `[[wikilinks]]` 自动建立知识网络 | 发现知识点之间的隐性关联 |

| **实战导向** | 注重裁判规则、证据认定、法律适用 | 律师实务工作 |
| **直接高效** | 拒绝客套话，直接给结论和依据 | 节省时间，提高工作效率 |

## 🚀 快速开始

### 环境检查清单

在使用前，确认以下事项：

- [ ] **WorkBuddy 已安装**，并创建了指向 `kb-wiki` 目录的工作空间
- [ ] **Obsidian 已安装**，并打开了 `kb-wiki` 仓库
- [ ] **目录结构已创建**（`raw/`, `wiki/`, `templates/` 等）
- [ ] **AGENTS.md 工作手册已创建**（律师专用版）
- [ ] **index.md 和 log.md 已创建**
- [ ] **MCP 服务已连接**（华宇元典、IMA、腾讯文档、企业微信）

如果以上任何一项未完成，请先参考《搭建方案》完成初始化。

### 第一次使用流程

#### 步骤1：放入第一批原始资料（5-10篇）

**推荐资料类型**：
- 最高人民法院指导性案例
- 你感兴趣的法律领域的核心法条
- 你正在研究的法律问题的学术文章
- 你参与或关注的真实案例的判决书

**采集方式**（详见"数据采集"章节）：
- 桌面端：使用 Obsidian Web Clipper 插件
- 移动端：分享到 IMA 知识库，再同步到 `raw/` 目录
- 手动创建：直接创建 Markdown 文件

#### 步骤2：执行首次编译

在 WorkBuddy 会话中输入：

```
阅读当前项目 raw/ 中新增的文件，按照项目根目录中的 AGENTS.md 的规范，
将它们编译到 wiki/ 目录中，并同时更新 index.md 和 log.md 文件。

本次编译范围：
- raw/articles/ 中的 5 个文件
- raw/judgments/ 中的 2 个文件

请按照 AGENTS.md 中的"页面类型规范"生成对应页面，并建立 [[wikilinks]] 关联。
```

#### 步骤3：检查编译结果

1. **查看生成的页面**：
   - 打开 `wiki/concepts/` 查看概念页
   - 打开 `wiki/entities/cases/` 查看案例页
   - 打开 `wiki/sources/` 查看源摘要页

2. **检查关联网络**：
   - 在 Obsidian 中打开关系图谱（Graph View）
   - 查看页面之间是否建立了连接
   - 如果发现孤立页面（没有连接），可以手动添加 `[[wikilinks]]` 或让 WorkBuddy 补充

3. **检查 index.md**：
   - 确认新页面已添加到对应分类下
   - 如果没有，手动添加或让 WorkBuddy 更新

4. **检查 log.md**：
   - 确认本次编译记录已添加
   - 记录中包含编译的文件和生成的页面

#### 步骤4：测试查询功能

编译完成后，测试查询：

```
根据知识库内容，解释"表见代理"的构成要件，并列出法条依据和相关案例。
```

**预期效果**：
- 小强会先读 `index.md`，定位相关页面
- 加载 `wiki/concepts/表见代理.md` 和相关案例页
- 直接给出结论和依据，不废话

#### 步骤5：根据效果迭代

如果编译结果不满意：
1. 修改 `AGENTS.md`，补充或修正规则
2. 删除 `wiki/` 中生成的页面
3. 重新执行编译命令

**记住**：`AGENTS.md` 是灵魂文件，需要迭代优化。

## 📥 数据采集

将法律资料采集到 `raw/` 目录中。

### 方式一：桌面端浏览器采集（推荐）

**适用场景**：在桌面浏览器中看到有价值的法律文章、判决书、学术论文等

**步骤**：

1. **安装 Obsidian Web Clipper 插件**：
   - 打开 Chrome/Edge 浏览器
   - 访问 Chrome 网上应用店
   - 搜索 "Obsidian Web Clipper"
   - 点击"添加至 Chrome"

2. **配置插件**：
   - 点击浏览器工具栏中的 Obsidian Web Clipper 图标
   - 设置保存位置：`kb-wiki/raw/` 目录
   - 建议按类型分子目录：
     - `raw/articles/` - 法律文章
     - `raw/judgments/` - 判决书
     - `raw/papers/` - 学术论文
     - `raw/regulations/` - 法律法规
     - `raw/notes/` - 个人笔记

3. **使用插件**：
   - 在浏览器中打开想要保存的网页
   - 点击 Obsidian Web Clipper 图标
   - 选择保存位置（子目录）
   - 点击"保存"
   - 插件会自动将网页内容抓取为 Markdown 格式，保存到指定目录

**优点**：
- 快速、简单
- 自动转换格式
- 保留原文格式和链接

**注意事项**：
- 部分网站可能有反爬虫机制，导致抓取失败
- 建议抓取后检查 Markdown 文件，确保内容完整
- 对于重要文档，建议同时保存 PDF 版本到 `raw/` 对应目录

### 方式二：移动端微信采集

**问题**：微信文章无法直接抓取

**解决方案**：使用 IMA 知识库中转到 `raw/` 目录

**步骤**：

#### 第一阶段：移动端收藏到 IMA

1. **安装 IMA 知识库 App**：
   - iOS：App Store 搜索"IMA"
   - Android：应用市场搜索"IMA"

2. **在微信中收藏文章**：
   - 打开微信文章
   - 点击右上角"..."菜单
   - 选择"复制链接"或"在 Safari 中打开"（iOS）
   - 粘贴链接到 IMA App，或直接在 IMA 中收藏

3. **或者直接分享到 IMA**：
   - 部分版本支持直接分享到 IMA 知识库
   - 具体操作见 IMA App 内说明

#### 第二阶段：从 IMA 同步到 `raw/` 目录

**方案 A：手动同步（推荐用于初期）**

1. 在桌面端打开 IMA 知识库
2. 找到收藏的文章
3. 导出为 Markdown 或 PDF
4. 保存到 `kb-wiki/raw/` 对应子目录

**方案 B：自动同步（使用 WorkBuddy 定时任务）**

1. **配置 IMA MCP 连接**：
   - 确保 WorkBuddy 已连接 IMA 知识库（已在你的配置中看到 `ima-mcp` 已连接）
   - 测试连接：在 WorkBuddy 中执行 `mcp__ima-mcp__get_knowledge_base_list` 查看知识库列表

2. **创建定时任务**：
   - 在 WorkBuddy 中创建自动化任务
   - 设置每周或每天执行一次
   - 任务内容：
     ```
     检查 IMA 知识库中的新增内容，将以下内容同步到 kb-wiki/raw/ 目录：
     1. 过去7天新增的文章
     2. 标签为"待整理"的内容
     
     同步时：
     - 保存为 Markdown 格式
     - 文件名格式：文章标题_YYYYMMDD.md
     - 保存到 raw/articles/ 目录
     - 在文件开头添加元数据：
       ---
       title: 文章标题
       source: IMA 知识库
       collected_at: YYYY-MM-DD
       tags: [标签1, 标签2]
       ---
     ```

3. **测试定时任务**：
   - 先手动执行一次，检查同步效果
   - 确认文件格式和保存位置正确
   - 调整任务参数（如同步频率、文件过滤规则等）

**优点**：
- 移动端采集通道打通
- 自动化同步，减少手动操作
- IMA 知识库可作为临时收集箱

**注意事项**：
- IMA 知识库的 API 可能有调用限制，注意控制同步频率
- 同步前先检查 `raw/` 中是否已有相同内容，避免重复
- 定期清理 IMA 知识库中的已同步内容

### 方式三：现有资料导入

**场景**：已有 Obsidian 或 flomo 笔记

如果你之前已经在 Obsidian 或 flomo 中积累了法律笔记，可以将它们导入到知识库。

#### 从 Obsidian 导入

1. **复制文件**：
   - 打开原 Obsidian 仓库目录
   - 选择要导入的 Markdown 文件
   - 复制到 `kb-wiki/raw/notes/` 目录

2. **批量处理（可选）**：
   - 如果文件很多，可以让 WorkBuddy 帮忙批量添加元数据
   - 示例命令：
     ```
     读取 raw/notes/ 目录中的所有 Markdown 文件，在每个文件开头添加以下元数据：
     ---
     title: 文件名（不含扩展名）
     source: 原有 Obsidian 笔记
     imported_at: 2026-06-27
     tags: [笔记]
     ---
     如果文件已有元数据（--- 格式），则合并而不是覆盖。
     ```

#### 从 flomo 导入

1. **导出 flomo 数据**：
   - 打开 flomo App 或网页版
   - 进入"设置" → "数据导出"
   - 选择导出格式（推荐 CSV 或 Markdown）
   - 导出所有笔记

2. **转换格式（如需要）**：
   - 如果导出的是 CSV，需要转换为 Markdown
   - 可以让 WorkBuddy 帮忙转换：
     ```
     读取 flomo_export.csv 文件，将每条笔记转换为独立的 Markdown 文件，
     保存到 raw/notes/ 目录。
     文件名格式：笔记标题_YYYYMMDD.md（如果笔记无标题，使用笔记前20个字符）
     在每个文件开头添加元数据：
     ---
     title: 笔记内容前50字符
     source: flomo
     created_at: 笔记创建时间
     tags: [从 flomo 标签提取]
     ---
     ```

3. **保存到 `raw/` 目录**：
   - 将转换后的 Markdown 文件保存到 `kb-wiki/raw/notes/` 目录

### 方式四：手动创建

1. 在 `raw/` 对应子目录中新建 Markdown 文件
2. 添加元数据（至少包含 title 和 source）
3. 粘贴或撰写内容

**元数据模板**：
```markdown
---
title: 文档标题
source: 来源（网址/书名/期刊名）
collected_at: 2026-06-27
tags: [标签1, 标签2]
---

# 文档标题

（正文内容）
```

### 数据采集原则

#### 宁缺毋滥

**原则**：只采集有价值的内容

**判断标准**：
- ✅ **应该采集**：
  - 权威来源（最高人民法院、高级人民法院、知名法学院校）
  - 经典案例（指导性案例、典型案例）
  - 前沿问题（新型案件、疑难问题）
  - 个人深度思考（不是简单的摘抄）

- ❌ **不应该采集**：
  - 质量低下的文章（错误观点、逻辑混乱）
  - 过时的法律法规（已被废止或修订）
  - 重复内容（已在知识库中的相似内容）
  - "先存了再说"的内容（没有明确价值）

#### 元数据完整

**原则**：每个原始资料都应该有基本的元数据

**最低要求**：
```markdown
---
title: 文档标题
source: 来源（网址/书名/期刊名）
collected_at: 采集日期
tags: [标签1, 标签2]
---
```

**推荐添加**：
- `author` - 作者
- `published_at` - 发布日期
- `type` - 类型（article/judgment/paper/regulation）
- `relevance` - 相关度（高/中/低）

#### 分类清晰

**原则**：采集时就按类型分类，便于后续编译

**推荐目录结构**：
```
raw/
├── articles/      - 法律文章、评论
├── judgments/     - 判决书、裁定书
├── papers/        - 学术论文
├── regulations/   - 法律法规、司法解释
├── notes/         - 个人笔记、思考
└── temp/          - 临时收集、待分类
```

## ⚙️ 知识编译

当 `raw/` 目录中有新增或更新的文件时，需要编译到 `wiki/` 目录。

### 编译命令模板

在 WorkBuddy 会话中输入：

```
阅读当前项目 raw/ 中新增的文件，按照项目根目录中的 AGENTS.md 的规范，
将它们编译到 wiki/ 目录中，并同时更新 index.md 和 log.md 文件。

本次编译范围：
- raw/articles/ 中的 X 个文件
- raw/judgments/ 中的 Y 个文件

请按照 AGENTS.md 中的"页面类型规范"生成对应页面，并建立 [[wikilinks]] 关联。
```

### 编译过程

WorkBuddy 会：

1. **读取 `raw/` 中指定的文件**
2. **识别法律概念、实体、法条、案例等**
3. **按照 `AGENTS.md` 规范生成 Wiki 页面**，保存到 `wiki/` 对应子目录
4. **使用 `[[wikilinks]]` 建立页面间关联**
5. **更新 `index.md`**（添加新页面链接）
6. **更新 `log.md`**（记录本次编译内容）

### 检查编译结果

编译完成后，检查：

1. **查看生成的页面**：
   - 打开 `wiki/concepts/` 查看概念页
   - 打开 `wiki/entities/cases/` 查看案例页
   - 打开 `wiki/sources/` 查看源摘要页

2. **检查关联网络**：
   - 在 Obsidian 中打开关系图谱（Graph View）
   - 查看页面之间是否建立了连接
   - 如果发现孤立页面（没有连接），可以手动添加 `[[wikilinks]]` 或让 WorkBuddy 补充

3. **检查 index.md**：
   - 确认新页面已添加到对应分类下
   - 如果没有，手动添加或让 WorkBuddy 更新

4. **检查 log.md**：
   - 确认本次编译记录已添加
   - 记录中包含编译的文件和生成的页面

### 重新编译

如果编译结果不满意，可以：

1. **删除生成的页面**（如果想完全重新编译）：
   - 删除 `wiki/` 目录中对应的页面
   - 或删除整个 `wiki/` 目录，重新编译

2. **增量编译**（如果只想编译新增文件）：
   - 在编译命令中指定范围：`raw/ 中新增的文件`
   - WorkBuddy 会跳过已编译的文件

## 🔍 智能查询

编译完成后，就可以在 WorkBuddy 中查询知识库了。

### 查询命令模板

#### 示例1：查询法律概念

```
根据知识库内容，解释"合同无效"的构成要件，并列出：
1. 法条依据
2. 相关概念
3. 典型案例
```

#### 示例2：查询案例

```
知识库中有没有关于"违约金调整"的案例？请列出案例名称和法院观点。
```

#### 示例3：查询法条

```
《民法典》关于合同履行的规定有哪些？请结合知识库中的案例说明。
```

#### 示例4：对比分析

```
请根据知识库内容，对比"合同无效"和"合同可撤销"的区别。
```

### 查询工作流

**小强**会按照 `AGENTS.md` 中定义的"查询工作流"执行：

1. **先读 `index.md`**：
   - 判断哪些页面与查询相关
   - 确定要加载的 Wiki 页面列表

2. **加载相关页面**：
   - 读取对应的 Wiki 页面内容
   - 不会直接读取 `raw/` 中的原始文档（除非必要）

3. **基于 Wiki 回答**：
   - 综合加载的页面内容回答用户问题
   - 引用时注明来源（页面名称）
   - **直接给结论和依据，不废话**

4. **知识沉淀（可选）**：
   - 如果查询产生了新的、有价值的知识
   - 小强会询问是否写入 Wiki，以便日后查询

### 查询技巧

1. **使用精确的关键词**：
   - ✅ "合同无效构成要件"
   - ❌ "那个什么无效的东西"

2. **指定查询范围**：
   - "根据知识库中的案例..."
   - "结合法条和案例..."

3. **要求列出来源**：
   - "请列出法条依据"
   - "请附上相关案例链接"

4. **迭代查询**：
   - 如果第一次查询结果不满意，可以追问
   - "能详细解释一下 XXX 吗？"
   - "有没有相关的案例？"

## 🔧 系统维护

知识库用久了会变乱，需要定期维护。

### 定期治理（每周一次）

在 WorkBuddy 中输入：

```
请对当前知识库进行一次健康检查，检查以下内容：
1. 有没有矛盾的信息（如同一个概念在不同页面中的定义不一致）
2. 有没有孤立的页面（没有任何其他页面引用它）
3. 有没有断掉的 [[wikilinks]]（指向不存在的页面）
4. 有没有过时的法条引用（如引用已废止的《合同法》）

检查完成后，生成一份健康报告，并给出修复建议。
```

### 手动维护

#### 添加缺失的关联

- 在 Obsidian 中打开某个页面
- 如果发现应该关联但未关联的概念
- 手动添加 `[[wikilinks]]`

#### 修正错误内容

- 如果发现某个 Wiki 页面内容有误
- 可以直接编辑 Markdown 文件
- 或在 WorkBuddy 中说："请修正 wiki/concepts/XXX.md 中的错误"

#### 删除无效页面

- 如果某个页面无用或重复
- 可以直接删除 Markdown 文件
- 然后更新 `index.md`，移除对应链接

### 更新 AGENTS.md

如果发现编译结果不稳定或不符合预期，可以更新 `AGENTS.md`：

1. 在 WorkBuddy 中打开 `AGENTS.md`
2. 补充或修正规则
3. 保存后，重新编译 `raw/` 中的文件

**常见更新场景**：
- 页面格式不一致 → 在"页面类型规范"中补充格式要求
- 关联不完整 → 在"编译流程"中强调关联建立规则
- 查询效果差 → 在"查询工作流"中补充查询策略

## ❓ 常见问题

### Q1：编译后查询效果不好，怎么办？

**可能原因**：
- `AGENTS.md` 中的查询工作流定义不清晰
- Wiki 页面质量差（内容不完整、关联缺失）
- `index.md` 中没有正确分类

**解决方案**：
1. 检查 `AGENTS.md` 中的"查询工作流"是否清晰
2. 检查生成的 Wiki 页面质量，必要时手动修正或重新编译
3. 更新 `index.md`，确保页面分类准确
4. 如果问题持续，可以让 WorkBuddy 重新编译，并优化 `AGENTS.md`

### Q2：如何判断知识库是否需要更新？

**判断标准**：
- 添加了新的原始资料到 `raw/`
- 发现了新的重要案例或法条解释
- `index.md` 中某个分类下的页面数量很少
- 查询结果不完整或不准确

**建议频率**：
- 每周编译一次新增资料
- 每月检查一次知识库完整性
- 每次遇到新案例或新法条，及时添加到 `raw/` 并编译

### Q3：Wiki 页面太多，不好管理怎么办？

**解决方案**：
1. **优化 `index.md`**：
   - 使用更细的分类（如按民法/刑法/诉讼法分类）
   - 添加标签或关键词

2. **使用 Obsidian 标签功能**：
   - 在 Wiki 页面中添加标签：`#民法 #合同`
   - 在 Obsidian 中通过标签快速筛选

3. **定期治理**：
   - 删除重复或无用的页面
   - 合并内容高度相似的页面
   - 检查并修复断链

### Q4：可以让 WorkBuddy 自动编译吗？

**可以**，使用 WorkBuddy 的定时任务功能：

1. 在 WorkBuddy 中创建自动化任务
2. 设置触发条件（如"每周一早上9:00"）
3. 任务内容：
   ```
   检查 raw/ 目录中过去7天新增的文件，
   按照 AGENTS.md 规范编译到 wiki/ 目录，
   并更新 index.md 和 log.md。
   ```
4. 保存并启用定时任务

**注意事项**：
- 自动编译前，确保 `raw/` 中的文件格式正确、内容完整
- 定期检查自动编译的结果，确保质量
- 如果编译出错，及时修正 `AGENTS.md` 或原始资料

### Q5：知识库可以分享给其他人吗？

**可以**，但需要注意：

1. **使用 Git 同步**：
   - 将 `kb-wiki` 目录推送到 Git 远程仓库
   - 其他人克隆仓库后，用 Obsidian 打开即可

2. **导出为静态网站**（高级）：
   - 使用 Obsidian 的"发布"功能或第三方工具（如 Jekyll）
   - 将 Wiki 页面转换为 HTML 静态网站
   - 部署到 GitHub Pages 或其他托管服务

3. **注意事项**：
   - 如果知识库中包含敏感信息（如客户信息、未公开案例），不要分享
   - 分享前检查 `raw/` 中的原始资料是否有版权问题
